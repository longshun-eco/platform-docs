## 名词说明和注释

**说明**

以下名词解释均针对应用/内购商品的购买过程。

|  |  |  |
| --- | --- | --- |
| 名词 | 说明 | 注释 |
| 新购 | 用户第一次购买或之前的购买行为到期后重新购买 | 用户没有当前要购买的商品的在用实例则购买行为视为新购。  **重要**  免费规格或不支持续费变配商品的试用规格购买后，再购买付费规格，视为新购。 |
| 续费 | 用户针对当前在用付费应用/内购实例进行提前延长使用周期的行为 | 对同规格版本、同使用人数的付费在用实例进行延长使用周期的购买行为。 |
| 升级 | 用户对当前在用的付费应用/内购实例进行高版本购买或者是更多使用人数的购买 | * 针对周期型商品付费规格之间从低规格到高规格的购买视为升级。 * 针对周期升级计量型商品规格版本从低到高或者使用人数从少到多均视为升级。 |
| 续费升级 | 用户对当前使用的规格做延长使用期的操作，延长部分的规格版本是升级后的高版本 | * 续费升级是在当前在用规格到期前操作。 * 续费升级操作成功后当前规格继续使用直到到期结束，结束后续费的周期段会升级到高规格版本。   **重要**  支持续费变配商品的试用规格购买后，在购买付费规格，视为续费升级。 |
| 续费降级 | 用户对当前使用的规格做延长使用期的操作，延长部分的规格版本是降级后的低版本 | * 续费升级是在当前在用规格到期前操作。 * 续费升级操作成功后当前规格继续使用直到到期结束，结束后续费的周期段会降级到低规格版本。 |

以上续费升级和续费降级统称为**续费变配**。

## **接入流程**

### **一、订阅消息推送事件**

推送消息接入分为**HTTP接入和RDS接入**，具体接入形式请在[开发者后台](https://open-dev.dingtalk.com/#/index)进行配置，配置样例如下。

* 推送类型为**使用HTTP推送**时，不需要做额外的订阅。

  ![[development-renewal-and-configuration-change-access_p265815.png]]
* 推送类型为**使用SyncHTTP推送**或**使用钉钉云推送**时，需要勾选下图标识出来的2个订阅事件。

  ![[development-renewal-and-configuration-change-access_p265822.png]]
* **订单推送、订购推送消息开发对接**

  **说明**

  钉钉开放平台消息推送分为HTTP模式和RDS模式。

  + HTTP模式是基于开发者后台配置的推送http地址将对应的推送信息通过post方式推送过去。
  + RDS模式是RDS实例接收钉钉消息时，在钉钉云环境下通过创建对应的数据库表结构，钉钉开放平台将推送的数据写入到指定的表。

  + 订单消息推送

    目前钉钉开放平台会在订单支付的时候，将对应的订单消息进行推送，推送形式当前为http模式和rds模式，两种模式推送过去的数据是一致的。

    推送的数据格式请参考[biz\_type=17](/document/development/data-format#section-5je-fzy-8p5)。
  + 订购消息推送

    目前钉钉开放平台会在订购服务开通、订购服务到期的时候，将对应的订购消息进行推送，推送形式当前为http模式和rds模式，两种模式推送过去的数据是一致的。

    推送的数据格式请参考[biz\_type=37](/document/development/data-format#eb0b7d60718ws)。

### **二、推送消息处理**

续费变配订单支付后的推送消息处理与普通订单有一定的区别，主要表现为：

1. 续费变配订单支付后有两个推送消息（bizType=17的订单支付推送消息，以及bizType=37的服务开通推送消息），而普通订单支付后仅有一个推送消息（bizType=17的订单支付推送消息）。
2. 收到续费变配订单的bizType=17支付消息后，仅需记录订单支付，无需开通对应服务（因为续费变配订单对应的服务未生效）。

   1. bizType=17消息中的orderType=RENEW\_UPGRADE时，订单为续费升级订单。
   2. bizType=17消息中的orderType=RENEW\_DEGRADE时，订单为续费降级订单。
3. 收到续费变配订单的bizType=37开通消息后，开通订单对应的服务。

   1. bizType=37消息中，syncAction=market\_service\_open且openType=3时，为变配续费服务开通消息。
   2. 可通过orderId将bizType=17消息与bizType=37消息关联。

### **三、联调和测试**

1. 进入自运营平台应用，选择商品上架，点击商机与服务，点击续费变配管理的编辑。

   ![[development-renewal-and-configuration-change-access_p530780.png]]
2. 设置自测组织。

   提前创建好测试组织，测试组织名称最后两个字必须是“测试”，例如，**xxx测试**。创建好之后，在下方页面，输入对应测试组织的corpId，点击保存进行报备。

   ![[development-renewal-and-configuration-change-access_p530784.png]]
3. 创建普通付费单。

   使用自测的组织在商品SKU页创建一个付费订单，并完成支付。

   **说明**
   * 不区分是新购/续费/升级订单
   * 必须是付费订单，免费和试用订单无法触发后续流程中的续费变配操作
4. 创建续费变配单。

   完成普通付费单支付后，回到商品SKU页，用自测组织创建续费变配单并支付。续费降级和续费升级的用户交互如下：

   ![[development-renewal-and-configuration-change-access_p530793.png]]

   **说明**
   * 续费升级时，在“选择周期”选项中，选中“剩余有效期”生成的订单是升级订单，选中开发者配置的周期生成的订单是续费升级订单。
   * 市场支持续费降级操作，但不支持降级。
5. 模拟续费变配生效的推送消息。由于续费变配订单的服务是在当前服务到期时才生效，为方便开发者测试和联调，这里提供了模拟续费变配订单生效消息推送的功能。

   在续费变配管理页面内，点击设置测试订单，将续费变配的订单号复制到下图红框中位置进行保存。

   ![[development-renewal-and-configuration-change-access_p530805.png]]
6. 续费变配生效验证。第5步设置测试订单保存成功后，开发者需要验证以下信息：

   * 开发者应用系统是否已经收到[biz\_type=37](/document/development/data-format#eb0b7d60718ws)的推送信息。
   * 开发者应用系统是否已经将客户版本变更为续费变配后的版本。

     **说明**

     因为是测试订单，所以钉钉会马上推送续费变配后的事件给到产品方案商，需要产品方案商应用系统立即自动将客户版本变更为续费变配后的版本，版本立即变更成功则表示测试成功。

     在正式环境下，钉钉不会立即推送续费变配后的事件给到产品方案商，会等到续费变配订单生效前再进行推送，这时产品方案商再将客户版本变更为续费变配后的版本内容即可。

### **四、提交审核**

联调和测试通过后，在续费变配管理页面点击提交审核，输入自测成功的信息，用于平台审核。

![[development-renewal-and-configuration-change-access_p530815.png]]

![[development-renewal-and-configuration-change-access_p530816.png]]

**说明**

提交审核后，平台会1-3工作日内完成审核，请关注页面审核结果，平台审核结果显示通过，则代表该第三方企业应用已支持续费变配功能。

![[development-renewal-and-configuration-change-access_p530821.png]]

## **常见问题**

**Q: 付费规格价格较高，测试联调需要支付的全部费用，成本太高，怎么办？**

答: 目前有以下几种方式测试。

方式一：正常付费后退款

测试联调的费用可七天无理由全额退款，若退款过程中有其他问题可以联系类目运营处理。

方式二：下单后申请订单改价

可以在自运营平台 -> 应用管理 -> 运营申请 -> 订单特价审批提交改价申请，待类目运营审批通过后生效。
