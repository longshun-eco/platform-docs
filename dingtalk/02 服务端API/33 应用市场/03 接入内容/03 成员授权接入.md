## 名词解释和注释

|  |  |  |
| --- | --- | --- |
| 名词 | 说明 | 注释 |
| 普通员工 | 企业组织中的非管理员用户，或者没有应用管理权限的管理员用户。 | - |
| 个人开通 | 组织下的普通员工授权自己开通应用。 | 个人开通属于个人行为，只会授权自己使用应用。 |
| 组织开通 | 组织下有应用管理权限的管理员授权组织开通应用。 | 组织开通属于组织行为，会授权组织信息和组织下的其他人使用应用。 |
| 成员授权 | 企业中的普通员工可以授权自己体验应用，并可邀请企业中的其他成员一同加入体验，成员授权后可以引导联系企业管理员为组织开通应用。 | 成员授权是应用市场原个人试用方案的2.0升级版本。 |

## 什么是成员授权

企业中的普通员工可以授权自己体验应用，并可邀请企业中的其他成员一同加入体验，成员授权后可以引导联系企业管理员为组织开通应用。让非管理员的关键人能直接体验到应用，并且可以在应用内相互协作。

当前除了成员授权方案，应用市场还存在其他两种针对个人的开通方案，分别为个人试用和个人版方案。成员授权方案从底层实现原理上看和个人试用方案更为类似（成员授权属于个人试用的升级2.0版本），在成员授权和个人试用中，员工开通都是基于用户当前的真实组织；而个人版方案则是基于为用户创建的的隐藏组织，为隐藏组织进行组织开通安装应用。

成员授权、个人版和个人试用的区别，请参见下方表格。

|  |  |  |  |  |  |
| --- | --- | --- | --- | --- | --- |
| 方案 | 非管理员应用开通实现 | 应用内邀请的快捷能力 | 付费商品默认开通的规格 | 应用使用数据 | 组织管理员对个人开通的感知和控制 |
| 成员授权（  个人试用的升级版） | 基于用户真实组织，为真实组织安装应用 | 分别提供了应用内邀请成员和邀请管理授权能力 | * **组织开通：**   如果提供免费规格且配置了优先开通免费规格，则会开通免费规格，其他情况开试用。   * **个人开通：**   如果提供了免费规格则默认开通免费规格，没有免费规格则开试用。  **说明**  免费规格的配置，见[免费规格接入](/document/development/free-specifications)。 | * 员工数据脱敏，相关开放接口在成员授权模式下存在接口受限情况。 * 企业下不同员工的应用使用数据集中在同一个真实组织中，可实现成员间应用内协作。 | * 应用列表页展示成员授权应用，并在应用设置中提供转组织授权、加入成员试用和停用应用能力。 * 可在应用列表页一键关闭成员授权。   **示例图片：** ![[development-document-on-authorization-of-application-market-members_7ff79665-195b-46ba-b4fe-524e8e1382e2.png]] |
| 个人版 | 通过普通员工在隐藏组织上开通应用实现 | 提供了联系管理员授权能力 | * **组织和个人开通：**   如果提供免费规格且配置了优先开通免费规格，则会开通免费规格，其他情况开试用。 | * 个人开通由于是给用户的隐藏组织安装应用，调开放接口用的是隐藏组织的信息，接口层面不受限。 * 企业下的不同员工开通会为不同的隐藏组织安装应用，数据天然隔离，成员间无法在应用内协作。 | * 无法感知员工开通了个人版。 * 可在应用设置页的应用使用范围中关闭员工体验，关闭后员工无法开通个人版。 |
| 个人试用 | 基于用户真实组织，为真实组织安装应用 | - | 组织开通和个人开通都只能开试用规格。 | * 员工数据脱敏，相关开放接口在个人试用模式下存在接口受限情况。 * 企业下不同员工的应用使用数据集中在同一个真实组织中。 | 无法感知，无法控制。 |

**重要**

接入了成员授权后，如果应用之前接入的个人试用或者个人版的影响：

1. 如果之前接入了个人试用，由于成员授权和个人试用在个人开通的底层实现上是基本一致的，存量客户可以做到无感切换，具体改造要点：

   1. 按照下文中**应用内引导邀请的接入**要求接入邀请功能。
   2. 按照下文中**开通事件处理**要求做改造，具体对于组织开通和个人开通的判断，在原有接入个人试用的基础上，注意开通规格的区别（成员授权可能开免费规格或者试用规格，个人试用都开试用规格，开放平台的biz=63的事件推送中有区分免费规格）。
2. 如果之前接入了个人版，存量客户的个人版应用可以从**工作台 > 我的**分组中进入，但是新客户的员工开通都会统一走成员授权方案。

## 开通事件处理

服务商对于个人开通和组织开通的判断需要通过钉钉的侧推送的**biz\_type=63**的事件做识别，biz\_type=63事件详情，请参考[数据格式](/document/development/data-format)。

bizType=63的推送事件中，有两个字段额外说明如下：

* tryoutType

  该字段表示此次开通是个人开通（personal\_tryout）还是组织开通（ enterprise\_tryout），同样也根据该字段判断此次开通后用户组织下该应用实例处于成员授权状态还是组织授权状态。

  例如，某组织中成员A首先进行了个人开通，此次推送的biz\_type=63事件中tryoutType = personal\_tryout，表明是个人开通，开通后当前应用处于成员授权状态；接着成员B也进行了个人开通，推送事件中tryoutType = personal\_tryout，开通后应用依然处于成员授权状态；最后管理员C进行了组织开通，推送事件中的tryoutType= enterprise\_tryout，此次开通后应用则处于组织授权状态。

  应用被组织授权后，在其应用使用周期内不会再转换为成员授权，过程是不可逆的（例如组织授权后个人/管理员再付费，推送事件中tryoutType均为enterprise\_tryout）。

  因此，服务商对于组织下当前应用实例状态（成员授权中还是组织授权中）的判断/更新，也可完全根据推送的tryoutType字段来判定即可，不需要结合订单事件。

  ![[development-document-on-authorization-of-application-market-members_p516259.png]]
* itemType

  该字段为开通的规格类型，注意当前只有对于付费商品的免费规格（"charge\_goods\_free\_item"）和付费规格（"charge\_item"）该字段一定存在。

  ![[development-document-on-authorization-of-application-market-members_p516260.png]]

当前成员授权已支持个人开通付费规格，组织中的成员（没有应用开通权限的）可在市场SKU页选择自助开通付费规格。但针对当前应用状态的不同，对成员付费也有限制（钉钉侧规则，会在入口拦截）：

* 当前应用未安装或处于成员授权中，且无其他成员已付费。此时组织中其他成员均可自主开通付费规格。开通后应用处于成员授权状态
* 当前应用处于成员授权中，且已有成员付费，只有应用使用范围中的成员可自助付费/续费/升级，付费后应用依然处于成员授权状态。
* 当前应用已组织授权，只有应用使用范围中的成员可自助付费/续费/升级，付费后应用依然处于组织授权状态。

对于付费规格开通，bizType=63事件返回中itemType = "charge\_item"，表明开通付费规格。且组织开通付费/个人开通付费均会推送biz\_type=63事件，对于付费开通后当前组织应用实例状态的判定/更新一样通过上述方式（推送事件中的tryoutType字段）来判定即可。

如果付费类型应用有免费规格，在应用市场和群内酷应用市场的授权开通中，会根据开发者后台规格管理中配置的优先开通顺序决定默认开免费还是试用规格。例如这里配置优先开通试用，而成员先行开通了免费规格，则组织开通后相当于进行了规格升级。配置了优先试用到期转免费的话，组织试用到期也会转组织免费，不受影响。

![[development-document-on-authorization-of-application-market-members_ec03c25e-549c-42d8-99a4-525c29bc4221.png]]

事件返回中的itemType = "charge\_goods\_free\_item", 用以区分付费商品的免费规格。

## 应用内引导邀请的接入

成员开通进入应用后应用内需要展示邀请横幅，横幅中会引导成员邀请其他同事或管理员进行授权。

**重要**

1. 这个邀请横幅展示只有在个人开通进入应用后才展示，企业开通不要展示这个横幅，个人开通和组织开通的判断，请参见上文**开通事件处理**。
2. 用户单击右上角“X”关闭横幅仅针对本次，下次登录此信息还会展示，只有转组织开通才不展示。

import { inviteToUseApp } from 'dingtalk-design-libs';

## 常见问题

### 1. 接入成员授权后biz\_type = 17的事件还会推吗？

答：看情况。biz\_type = 17 和 biz\_type=63 是两个独立的事件没有必然的联系。biz\_type = 17是订单事件，只有新产生订单才会推，而接入成员授权后成员开通不一定会产生订单（例如不同成员在同一天开通试用规格，由于试用周期一致就不会再产生订单），而 biz\_type=63 则是每次成员授权或组织授权都会触发。所以要区分出本次开通后应用状态是成员授权还是组织授权，需要根据biz\_type=63 事件判断。

### 2. 普通成员开通试用后，管理员还能开通企业试用吗？

答：可以的。并且会推biz\_type=63 的事件。

### 3. 管理员开通组织授权后对原来成员开通的有什么影响？

答：首先，组织授权后应用状态由成员授权变成了组织授权。对于原成员的使用，要看管理员在开通时选择的使用范围，相当于重置了使用范围。不在使用范围中的成员，钉钉侧就不会提供入口进入应用。

### 4. 管理员开通企业后还会有个人在同一组织上开成员吗？

答：不会。当前如果处于组织授权状态，就不会再允许个人开通。

### 5. A成员授权后，A是否可以停用应用？如果可以是否有对应的事件？

答： A作为成员不可以停用应用，但是A可以退出使用。正常情况下A退出使用就是将A从应用的使用范围中移除了，但是如果A是最后一个处于应用使用范围中的人时，这时候会触发应用的解除授权，会推送应用的解除授权事件。

### 6. A成员授权且到期后，是否可以二次成员授权开通应用？如果不可以，是钉钉侧直接拦截还是需要我们配合处理？

答：到期后不能再次开通，钉钉侧会有拦截。

### 7. 企业管理员已授权开通主应用，成员A不在应用可见范围，A是否可以成员授权开通应用？如果不可以，是钉钉侧引导还是需要我们配合处理？

答：A不可以成员授权开通应用，钉钉侧会引导A联系管理员将他加入到可见范围。

### 8. 成员开通试用规格的前提下，例如商品试用期是15天，多个成员开通每个人的试用周期都是单独计算15天还是公用一个周期？

答：每个成员都有独立的15天。例如组织下两个成员A和B，成员A在1号开始试用，则他自己会在15号到期（到期后钉钉侧会将A从应用使用范围中移除），成员B在5号的时候开始试用，则B会在20号到期并移除使用。另外B在15号开通的时候会下订单，订单有效期从15号到20号。

## 验收

### 验收前自测

1. 进入自运营平台，进入**应用管理 > 商品上架 > 选择需要测试的应用 > 体验授权 > 成员授权设置。**

   ![[development-document-on-authorization-of-application-market-members_p485369.png]]
2. 在**成员授权设置**页面，单击**设置自测组织**进行配置。

   ![[development-document-on-authorization-of-application-market-members_p485391.png]]
3. 在**报备测试组织**页面，输入**测试组织的CorpId**后，单击**保存**按钮。

   ![[development-document-on-authorization-of-application-market-members_p485393.png]]

**说明**

1. 测试组织的名称需要以**测试**字样结尾。
2. 每次报备一个，如果该测试组织试用完无法再测试，可在提交审核前再保存一个，同时之前报备的将失效。

**重要**

测试的前提是应用必须在市场上架（上架包活自有渠道和应用市场推广两种），有些服务商可能需要在自己的测试环境下测试流程，如果是未上架的测试应用是不行的。如果觉得线上测试的风险不可控，可以把测试的应用提一下商品上架，然后找对接的类目运营说明情况给配置成以自有渠道推广的方式上架。上架后就能用测试应用测试流程了，相当于重新搞个供测试市场流程的品，这个流程相对繁琐一些，所以建议风险可控的情况下直接用正式应用。

### 提交验收审核

1. 进入自运营平台，进入**应用管理 > 商品上架 > 选择需要测试的应用 > 体验授权 > 成员授权设置**。

   ![[development-document-on-authorization-of-application-market-members_p485369.png]]
2. 如果没有设置测试组织要先设置，设置完成状态应为**待送审**状态。

   ![[development-document-on-authorization-of-application-market-members_p477320.png]]
3. 单击**提交审核**，**上传自测视频**（可上传多个文件，视频文件建议不要太大，否则影响上传速度）。

   ![[development-document-on-authorization-of-application-market-members_p477321.png]]
4. 提交审核后，状态会变为**审核中**，此时等待审核人员审核即可。如果审核拒绝，状态会展示变为**审核拒绝**并会给出拒绝原因，修复完成之后可再次提交验收审核。如果审核通过，功能就自动全量上线了，会正式发布到线上（广场详情页和工作台推荐）。

   ![[development-document-on-authorization-of-application-market-members_p477324.png]]

### 验收视频材料用例说明

1. **成员开通**

* 用测试组织下的**非管理员A**在应用市场的商品详情页中开通应用（非市场推广应用通过推广码的形式进到详情页），开通后进到应用内，预期展示引导邀请的横幅。
* **非管理员A**开通进入应用内后，通过单击**邀请同事**，会弹出两个选人列表。分别是**通过群成员清单选择**和**通过组织通讯录选择**。通过列表选择到具体的员工并单击确认后，会返回提示**邀请成功**，被邀请人接收到单聊消息卡片，并且可在卡片上授权个人开通。

  ![[development-document-on-authorization-of-application-market-members_p477283.png]]
* 非管理员A单击**管理员授权**，会弹出管理员列表，选择管理员后，会给管理员发送单聊卡片消息，并且可在卡片上授权企业开通。

  ![[development-document-on-authorization-of-application-market-members_p476804.png]]
* **非管理员A**推出应用再次进入应用内，预期展示邀请横幅。

2. **管理员开通**

* 用测试组织下的管理员B在应用市场的商品详情页中开通应用，开通后进到应用内，预期不展示邀请横幅。

将上述**成员开通**和**管理员开通**的步骤录制成两个视频，提交即可。
