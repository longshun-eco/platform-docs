更新于 2026-01-22在调用钉钉开放平台接口时，开发者会收到对应的返回结果，包含成功或失败的错误码信息。通过分析错误码，可快速定位问题并进行调试。

本错误码体系适用于以下应用类型：

* **企业内部应用**：由企业自行开发并仅供本企业使用的应用。
* **第三方企业应用（ISV应用）**：由独立软件开发商（ISV）开发并提供给多个企业使用的服务。
* **第三方个人应用**：产品方案商开发者开发，提供给钉钉上个人用户使用的应用。

**重要**

应用程序可根据errcode来判断出错的情况，不要依赖errmsg来匹配，因为errmsg可能会调整。

## 权限说明

不同角色对API的调用权限存在差异，请注意以下限制：

* **企业管理员**：具备完整的接口调用权限，可配置应用权限、管理成员与部门，并获取所有数据。
* **普通成员**：仅能访问被授权的应用和接口，部分敏感操作（如创建部门、发送全员消息）受限。

若遇到权限相关错误（如 `errcode=60011`），请前往[钉钉开放平台](https://open-dev.dingtalk.com/#/index) > 应用详情页 > **权限管理** > 添加接口权限，勾选所需能力。

## 术语定义

在错误码体系中，涉及以下关键字段，其含义如下：

* `errcode`：主错误码，用于程序判断请求是否成功。值为 `0` 表示成功；非 `0` 表示失败，需结合 `errmsg` 或 `sub_code` 进一步排查。
* `sub_code`：子错误码，提供更细粒度的错误分类，通常配合 `errcode=88` 使用，用于区分鉴权异常的具体原因。
* `sub_msg`：子错误信息，辅助开发者理解具体错误场景，内容可能随版本调整，不建议作为程序匹配依据。
* `errmsg`：错误提示信息，描述错误原因，主要用于人工阅读和初步排查。
* `access_token`：获取的应用访问凭证。

## 错误码格式说明

* 当`errcode = 0`时，表示接口调用成功。示例：

  ```json
  {
      "errcode": 0,
      "access_token": "xxx",
      "errmsg": "ok",
      "expires_in": 7200
  }
  ```
* 当`errcode = 88`时，表示请求失败，需关注返回结果中的`sub_code`和`sub_msg`字段。此类情况通常由`access_token`不合法或缺少接口调用权限引起。示例：

  **说明**

  即使确认`access_token`正确，也请核对当前应用的`appid`是否配置正确。

  ```json
  {
      "errcode": 88,
      "sub_code": "60011",
      "sub_msg": "没有调用该接口的权限",
      "errmsg": "ding talk error[subcode=60011,submsg=没有调用该接口的权限]",
      "request_id": "5imjny7u9xwa"
  }
  ```
* 当`errcode`为其他非零值时，表示请求失败，需重点关注`errmsg`字段内容。示例：

  ```json
  {
      "errcode": 40078,
      "errmsg": "不存在的临时授权码"
  }
  ```
