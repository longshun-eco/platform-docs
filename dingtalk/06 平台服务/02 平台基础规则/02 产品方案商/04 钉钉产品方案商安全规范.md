## **一、第三方企业应用安全认证评分标准**

### **第一章 目的和适用对象**

#### **1.1 目的**

为营造规范、有序、安全的钉钉开放平台环境，提升钉钉开放平台应用的安全能力，保障钉钉平台开发者及用户的权益，特制定本定级评分标准。

#### **1.2 适用对象**

本标准适用于钉钉开放平台第三方企业应用的安全定级评分工作。本期暂适用于以下第三方企业应用类型，具体根据业务情况确定。

|  |  |  |
| --- | --- | --- |
| 应用分发方式 | 业务场景 | 安全定级评分要求 |
| 上架到应用广场 | 上架到钉钉应用广场，可被钉钉用户搜索。 | 须在应用上架前确定安全保护层级同时进行安全评估，并由平台确认应用的安全分。 |
| 上架自有渠道推广 | 自有渠道线下拓客，安装企业数量不受限制。 | 须在应用上架前进行安全评估，暂无需由平台确认应用的安全分。 |

### **第二章 术语和定义**

|  |  |
| --- | --- |
| 术语 | 定义 |
| 敏感类目 | 指钉钉参考国家或行业相关的安全标准关注的重点行业，结合类目数据的重要性和敏感程度划分的类目，例如数字教育、人事管理、财税管理、商旅出行、数字政务、数字医疗、安全等。 |
| 敏感API | 指钉钉根据Open API提供数据的敏感程度进行划分的API分类，例如可获取个人敏感信息（如手机号、身份证图片、通讯信息、地址信息、金融账号信息等）、企业敏感数据（如企业的人事主数据、完整的组织架构数据等）。 |
| 安全保护层级 | 指钉钉用于实施不同安全要求策略划分的不同安全级别。 |
| 应用定级 | 指钉钉根据第三方企业应用的敏感程度和重要程度划分到相应的安全保护层级。 |
| 基础要求项 | 指应用在需要上架的情况下，需要满足的关键安全要求项。 |
| 计分要求项 | 指基础要求项之外，计算应用安全性得分的要求项。 |
| 加分要求项 | 指在基础要求项和计分要求项之外，应用满足评分项情况下能额外加分的要求项。 |
| 准入评估 | 指钉钉对第三方企业应用上架前进行的安全性评估。 |
| 年度评估 | 指钉钉每年对已上架的应用进行的安全性评估。 |

### **第三章 定级和评分**

#### **3.1 安全保护层级**

根据第三方企业应用所属的类目、调用开放平台的API、用户月活数量/开通组织数，第三方企业应用层级分为以下三级。

|  |  |
| --- | --- |
| 层级 | 说明 |
| 第一级（S1） | 不满足属于敏感类目、调用敏感API、用户月活数/开通组织数≥10000三者任一的应用。 |
| 第二级（S2） | 满足属于敏感类目、调用敏感API、用户月活数/开通组织数≥10000三者其一的应用。 |
| 第三级（S3） | 满足属于敏感类目、调用敏感API、用户月活数/开通组织数≥10000三者其二及以上的应用。 |

#### **3.2 定级对象**

|  |  |  |
| --- | --- | --- |
| 章节 | 定级对象内容 | 说明 |
| 3.2.1 | 定级要素 | 第三方企业应用安全保护层级的定级要素包括：   * 所属应用类目。 * 调用OpenAPI。 * 用户月活数/开通组织数。 |
| 3.2.2 | 定级标准 | 应用安全保护对象的级别由不同外在表现综合决定。定级对象的定级标准归结为以下三种：   * 是否为敏感类目。 * 是否调用敏感OpenAPI。 * 用户月活数/开通组织数是否大于等于1万。 |

#### **3.3 定级标准和钉钉第三方企业应用安全保护层级的关系**

|  |  |  |  |
| --- | --- | --- | --- |
| 层级 | 敏感类目 | 敏感OpenAPI | 用户月活数/开通组织数≥1W |
| S1 | ❌ | ❌ | ❌ |
| S2 | ✅ | ❌ | ❌ |
| ❌ | ✅ | ❌ |
| ❌ | ❌ | ✅ |
| S3 | ✅ | ✅ | ❌ |
| ✅ | ❌ | ✅ |

#### **3.4 评分标准**

|  |  |
| --- | --- |
| 评分标准 | 说明 |
| 基础要求项 | 基础要求项的满足率不低于80%且安全评分得分高于60分，即可通过应用上架的安全审核。   * S1、S2、S3三种不同安全保护层级的基础要求项安全要求有所不同。安全保护层级越高，基础要求项越多，应用的安全分越高，合作过程中可获取权益的机会也将越多。 |
| 总分 | 总分100分制。   * 满足基础要求项安全要求为60分。 * 其他计分要求项通过权重计算加上加分要求项总分为100分。 |

#### **3.5 定级评分流程**

第三方企业应用的安全保护层级定级根据应用所属的类目、调用的敏感API、用户月活数/开通组织数三个维度进行检测判断，定级操作由钉钉开放平台完成应用的检测定级，无需第三方企业应用介入。定级评分流程图如下图所示。

### **第四章 定级检测**

由钉钉开放平台检测应用所属的类目、调用的敏感API、用户月活数/开通组织数，确定应用的安全保护层级。

|  |  |
| --- | --- |
| 定级要素检测 | 定级标准检测 |
| 敏感类目检测 | 钉钉开放平台检测判断应用的所属类目是否为敏感类目，例如数字教育、人事管理、财税管理、商旅出行、数字政务、数字医疗、安全等类目。 |
| 敏感API调用检测 | 钉钉开放平台检测判断应用是否调用了开放平台的敏感API，例如含有个人敏感字段（如身份证号、手机号、身份证图片等可识别个人身份的字段，或通讯信息、地址信息、金融账号信息等非可识别个人身份敏感信息）和企业敏感数据（如企业的人事主数据、完整的通讯录架构数据等）。 |
| 用户月活数/开通组织数检测 | 钉钉开放平台检测判断应用的用户月活数或已开通的组织数是否大于等于1万。 |

### **第五章 安全评估**

安全评估分为安全准入评估和安全年度评估。

#### **5.1 安全准入评估**

|  |  |
| --- | --- |
| 安全评估内容 | 说明 |
| 安全准入评估 | 在应用上架前，应用填写安全准入评估的内容项，其中基础要求项为核心满足的安全要求，计分要求项为可选的满足项要求，根据准入评估的填报内容，钉钉平台侧计算应用安全得分。满足基础要求项的安全要求即通过安全审核，可上架。 |
| S1、S2和S3不同层级的准入评估安全检测项，请参见[准入评估检查项](/document/services/entry-assessment-check-items)。  准入评估的填写内容请参见[安全准入评估指南](/document/services/security-access-evaluation-guide)。 |

#### **5.2 安全年度评估**

|  |  |
| --- | --- |
| 安全评估内容 | 说明 |
| 安全年度评估 | 钉钉在准入评估的安全要求项基础上，阶段评估将增加服务商在应用安全运营上的评估。阶段评估将对应用的安全分进行重新计算，安全分满足上架安全要求的应用将限期整改，未按期完成整改的应用将按照运营规范下架整改。 |
| 阶段评估的安全运营域的安全要求项默认满足，不满足情况下扣除安全要求项的得分。在年度的阶段评估中，根据服务商在年度内安全运营域的响应和处理情况，可获得安全分的加分。 |
| 安全年度评估的要求项，请参见[年度评估检查项](/document/services/annual-assessment-inspection)。 |

### **第六章 安全分计算**

#### **6.1 准入评估**

|  |  |
| --- | --- |
| 评分项类别 | 分数计算规则 |
| 基础要求项计算 | 基础要求项为应用上架关键满足的安全要求项，满足所有基础要求项情况下得分为60分，若基础要求项满足率低于80%，则直接不通过。 |
| 计分要求项计算 | 计分要求项得分=（所有满足的要求项总得分/所有计分要求项的总分）\*30，计分要求项最高得分为30分。 |
| 加分要求项计算 | 在基础要求项和计分要求项总分90分之外，提供的独立加分项，总分为10分。 |
| 准入评估安全分总分 | 安全分总分=基础要求项得分（即基础的60分）+计分要求项得分（即按需满足的30分）+加分要求项得分（即独立加分的10分）。    **重要**  若基础要求项满足率低于80%或总分低于60分则评估不通过，不能上架，即通过安全评估的应用安全分需满足60分≤安全分≤100分。 |

#### **6.2 年度评估**

在基础要求项+计分要求项+加分要求项计算的安全总分基础上，增加年度安全运营分计算。

|  |  |
| --- | --- |
| 年度评分项类别 | 分数计算规则 |
| 年度安全运营分计算 | 根据服务商在年度内安全运营域的响应和处理情况，计算安全运营分的得分，运营分在年度安全评估的总分占比为20%。 |
| 年度评估安全分总分 | 安全分总分=（基础要求项得分+计分要求项得分+加分要求项得分）\*80%+年度安全运营分得分\*20%。    **重要**  若基础要求项满足率低于80%或总分低于60分的需按期进行安全整改，未完成整改的将按照规范进行处罚，例如下架应用，即通过安全评估的应用安全分需满足60分≤安全分≤100分。 |

### **第七章 确认安全分**

安全分的最终结果将在开发者后台中透出给到应用开发者，且在应用开通的产品详情页透出展示，作为应用安全性认证的评估标准透出给到开通应用的企业组织客户。

**说明**

安全分暂不对外透出，前期将通过钉钉的应用开发助手（或钉钉小秘书），将安全分的结果推送给提交评分资料的应用开发者，待后期产品功能和运营机制完善后，会在开发者后台展示给应用开发者，并在应用开通的产品详情页展示。

### **第八章 变更说明**

正常情况下，钉钉会每年定期对第三方企业应用进行安全评估，并调整安全保护层级。其他特殊情况下（如安全突发事件），钉钉会根据《钉钉产品方案服务商管理总则》《钉钉产品方案服务商违规处罚规范》等相关要求及影响程度进行相应的安全保护层级调整。

## **二、日常安全运营规范**

### **第一章 目的和适用范围**

#### **1.1 目的**

为助力钉钉的产品方案商（下称“ISV”、“服务商”或“合作伙伴”）为钉钉上的客户提供更加安全可靠的SaaS应用服务，提升钉钉开放平台第三方应用服务的安全水位，保障钉钉平台上开发者及用户的权益，特制定此规范。

本规范指钉钉开放平台产品方案商在日常经营中，需要承担对上架商品的安全性进行保障的职责，及时治理钉钉开放平台检测发现的安全风险。针对未按照规范进行安全风险治理的行为，钉钉开放平台将根据规范要求进行惩戒处罚。

#### **1.2 适用范围**

适用于已在钉钉开放平台上架应用的产品方案服务商，包括上架应用（含自有渠道推广、应用市场推广）、甄选、精选商城、套件（应用部署在钉钉环境除外）等业务合作模式的应用。

### **第二章 术语和定义**

|  |  |
| --- | --- |
| 术语 | 定义 |
| 上架应用 | 指产品方案商通过钉钉开放平台上架提供服务的SaaS应用，包括自有渠道推广和应用市场推广的SaaS应用。 |
| 安全认证 | 指上架应用根据《第三方企业应用安全认证评分标准》在通过安全准入评估或安全年度评估认证后，由平台确认的安全得分后，计算得出优/良/中等不同安全认证等级的勋章标签。 |
| 风险延期 | 指钉钉平台推送是风险通知后，未按照治理时间要求完成风险的处理情况下，及时反馈延长处理时间。 |
| 风险超期 | 指推送的风险在延期一次后，仍未按照延期后的时间要求完成风险的处理。 |
| 超期定级 | 指风险处理超期后未完成处理，结合风险危害判定风险级别，分为R0、R1、R2、R3、R4等级别，对应安全违规事件的S0、S1、S2、S3、S4等级别。 |
| 超期处罚 | 指风险处理超期后，根据风险的超期定级的级别，执行相应的惩戒措施。 |

### **第三章 规范细则**

* 风险通知：钉钉平台将通过人工、通知群、工单流程、卡片通知等不同方式推送风险治理的通知；
* 风险处理：ISV在收到钉钉平台风险治理通知后，应在治理时间期限内完成相应风险的治理动作，并反馈处理结果；
* 风险关闭：风险治理的完结状态，以钉钉平台验证确认风险已有效治理为准，关闭时间以钉钉平台确认风险实际完成治理的时间为准；
* 风险延期：无法在治理时间内完成风险处理的情况下，原则上只允许ISV申请一次风险延期，延长风险处理的时间；
* 风险超期：风险延期一次后，仍未完成风险的治理，钉钉平台则判定为该风险超期；
* 超期定级：风险超期后，将按照风险定级处理，具体请参考“第四章 治理风险项明细说明”；
* 超期处罚：风险超级定级为R0~R4级别对应安全违规事件的S0~S4级别，若定级为R0~R3，则遵循《钉钉产品方案服务商违规处罚规范》的执行处罚措施，若定级为R4级别，季度内累计超期次数≥3次则升级为R3级别，季度内累计超期次数≥5次则升级为R2级别。

## **三、安全违规事件管理规范**

### **第一章 目的和适用对象**

#### **1.1 目的**

为营造安全、规范、有序的开发环境，保障开发者（亦称“产品方案服务商”）及其用户（即开发者通过应用进行服务的钉钉平台其他用户）的合法权益，特制订此规范。

本规范系对《钉钉产品方案服务商违规处罚规范》的有效补充。本规范与《钉钉产品方案服务商违规处罚规范》有冲突的，以本规范为准。

#### **1.2 适用对象**

适用于钉钉开放平台（以下简称“开放平台”）所有产品方案商及其所开发对钉钉平台用户提供服务的应用。

### **第二章 术语和定义**

|  |  |
| --- | --- |
| 术语 | 定义 |
| 事件级别 | 指钉钉平台根据违规内容的性质、危害和影响等因素判断事件的严重程度级别，包括严重（S0）、高危（S1）、中危（S2）、低危（S3）、普通（S4）等共五个级别。 |
| 事件类型 | 指钉钉平台根据违规内容判断事件所属的类别，目前主要包括恶意违规且情节严重、安全风险隐患、数据安全、应用安全等共四个类别。 |

### **第三章 规范细则**

* 违规来源：违规事件的来源主要包括钉钉平台主动检测排查、行政管理部门的通报或通知、司法机关的法律文书、用户或客户的投诉举报、新闻媒体的报告、其他合法合规的渠道。
* 违规判定：钉钉平台将根据本规范的“第四章 安全违规事件”相关内容进行判定违规事件的级别和类型。
* 违规处罚：安全违规处罚遵循《钉钉产品方案服务商违规处罚规范》的处罚措施，S0级别事件按照一类违规处罚，S1级别事件按照二类违规处罚，S2级别事件按照三类违规处罚，S3级别事件按照四类违规处罚；S4级别事件，季度内累计超期次数≥3次则升级为S3级别，季度内累计超期次数≥5次则升级为S2级别。
* 违规决定的做出：遵循《钉钉产品方案服务商违规处罚规范》的“【决定的做出】”。
* 违规通知：遵循《钉钉产品方案服务商违规处罚规范》的“【通知】”。
* 违规申诉：遵循《钉钉产品方案服务商违规处罚规范》的“【处罚措施解除与申诉】”。

### **第四章 安全违规事件**

|  |  |  |  |

| **事件级别** | **事件类型** | **序号** | **违规事件** |

| 2. 安全风险隐患事件 | 2.1 | 企业自身问题引发的安全风险隐患，可能给用户、钉钉或阿里巴巴造成重大损失，或恶劣影响。 |
| 3. 数据安全事件 | 3.1 | 开发者将其在开放平台开通相关账户信息向任何第三方进行转让、出借、出租等其他原因被第三方使用。 |
| 3.2 | 开发者将其在开放平台所获得的信息数据用于非合作范围和目的，或者未经开放平台同意情况下以任何形式直接或间接地披露、泄露、共享、提供给第三方。 |
| 3.3 | 开发者将钉钉要求存储在阿里云计算巢服务上的数据，违规转移到计算巢服务的可信环境之外的其他环境存储。 |

| 3. 数据安全事件 | 3.4 | 开发者未履行《安全责任共担及安全承诺》中的责任和义务，未妥善保护钉钉用户的数据安全导致数据严重泄露。 |

| 4. 应用安全事件 | 4.1 | 开发者的应用恶意干扰或企图干扰开放平台、其他开发者应用或其任何部分或功能的正常运行，如对钉钉平台或平台上的应用进行恶意的安全攻击。 |
| S3 级别 | 2. 安全风险隐患事件 | 2.3 | 开发者长期未响应或处理钉钉安全巡检发现的风险隐患。 |
| 3. 数据安全事件 | 3.6 | 开发者的应用在发生严重数据泄露事件，在收到通知的24小时内未响应和及时止血处理。 |
| 3.7 | 开发者未遵循合理和必要性基本原则，过度申请获取用户权限和数据，或滥用已申请权限和数据，且在收到平台推送的治理通知后仍未按照要求落实治理工作。 |
| 3.8 | 开发者的应用在需要收集、变更用户个人数据时，没有事先获得用户的授权同意，或没有明确地告知用户其相关数据收集的目的、范围及使用方式。 |

| 4.3 | 开发者应用系统未按照要求满足安全部署方案的管控，且在收到平台整改通知后仍未按照要求落实整改工作。 |
| 4.4 | 开发者应用系统发生安全入侵类故障，例如导致用户核心数据被恶意篡改泄露传播、应用内容被非法篡改且内容危害性大、因应用漏洞被入侵被大面积传播讨论。 |
| S4 级别 | 2. 安全风险隐患事件 | 2.4 | 开发者在收到严重高危漏洞事件预警通知后，未按照要求进行自查和反馈。 |
| 2.6 | 开发者屏蔽钉钉平台IP，规避钉钉日常服务及安全的排查。 |
| 3. 数据安全事件 | 3.12 | 开发者在未经开放平台同意情况下通过名下多个（不同）应用对要求部署在阿里云计算巢服务上的应用服务发起API请求。 |
| 3.13 | 开发者的应用不支持客户数据批量导出或导出数据缺失、不支持客户删除数据导致投诉。 |
| 4. 应用安全事件 | 4.5 | 开发者对外泄露了应用系统的高危端口、管理接口或源码目录等敏感信息，造成对从钉钉获取的数据存在严重泄露的风险。 |
| 4.6 | 开发者的应用中出现账号提权、安装后门程序等主动危害系统安全的行为。 |

| 4.9 | 开发者未按要求对安全类异常或风险问题及时修复和反馈。 |

### **第五章 变更说明**

正常情况下，钉钉会每年定期对本规范进行调整和修订，新版本发布后试运行一个月，在试运行期间有任何问题可以提交工单或联系钉钉平台进行反馈和提供建议。
