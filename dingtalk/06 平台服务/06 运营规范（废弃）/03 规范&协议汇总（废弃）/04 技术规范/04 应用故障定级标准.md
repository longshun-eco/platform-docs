## 故障定义

在日常运营中，无论什么原因，产品出现服务中断、服务品质下降，导致用户体验下降的情况，都称为故障（故障不包括用户方环境引发的场景）。

## 制定故障定义等级的原则

通过故障等级的定义，可分析出：故障等级定义的4个关键点。制定故障等级定义的原则主要围绕这4点。

基础：**要以用户体验为核心**

重要：就高原则共识、标准化、可落地运营

|  |  |
| --- | --- |
| **原则** | **详情** |
| 就高原则共识 | 1.影响多个产品线的故障采用就高原则。  2.原则上故障定级需要与直接依赖业务方达成一致（无法达成一致的采用就高原则）。 |
| 标准化，用数据定级 | 1.业务模块全部可以被包括并定义，拆分到最细的功能模块。  2.定级指标要量化，通过监控的实时数据来衡量，例如“P2故障， 首页访问成功率低于90%”。 |
| 可落地运营 | 1.上架应用必须接入钉钉开放平台稳定性监控。  2.上架应用须制定用户反馈的渠道，建立监控机制，可以发现未覆盖的故障。  3.建立完善的故障恢复机制，在第一时间恢复，保证用户体验。 |

## 故障等级定义

故障等级定义内容：由业务模块、影响面，故障P等级，三部分组成。P等级是故障等级的基础序列（priority level），故障处理的综合优先级。

以用户体验的角度来整体衡量故障，通过业务模块的**“****重要性****”****，“****影响面大小****” “****处理时长****”**决定故障等级。

1.业务模块

分为“核心模块”、“次核心模块“与”“非核心模块”。

3.故障等级

根据业务模块和影响面，每种故障处理级别也不同。故障从P1—P4 ， P1最严重，P4最轻。

| **P2** | 重大故障，需要通知到钉钉，并联系钉钉研发，处理故障 |

同样等级的故障，但引起较大舆情的，参考S级和舆情故障定级（见下文），采用就高原则处理。

## 可用性定义

**可用性事件**：页面或接口失败率>60% 且持续15分钟以上。

**不可用性时长：**计入可用性事件的扣减运维不可用时长。示例场景1: 同一故障累计不可用超过15分钟，但中间有恢复，单次不可用未持续超过15分钟，则不算可用性事件，不扣减运维不可用时长； 示例场景2:同一故障出现多次下跌，中间有短暂恢复，但其中有单次下跌超过15分钟，则多次下跌的累加不可用时长作为本次不可用性事件的扣减运维不可用时长。

**全年不可用时长**：按照应用99.99%运维SLA标准，全年允许的不可用时长为53分钟。年度不可用时长累计时间范围为财年4月1日到次年3月31日。

## S级和舆情故障定级

针对有用户投诉的故障，故障恢复后3h内，钉钉客服团队会提供用户投诉反馈至钉钉的列表（包含用户id），服务商提供投诉反馈至服务商的列表（包含用户uid），综合数据后，运营和BI将用户id转化成企业id，并提供企业等级（大客户or普通客户），SLA：2小时。
